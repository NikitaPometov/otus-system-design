Система использует несколько типов хранилищ в соответствии с архитектурой
микросервисов и требованиями по задержкам, доступности и объему данных.
Данные разделены по доменам сервисов и не шарятся на уровне БД.

==== Реляционные БД (PostgreSQL)

[arabic]
. Сервис Управления Пользователями
.. Тип: PostgreSQL. Назначение: хранение профилей пользователей и ограничений
(бан/разбан). Необходимый объем: ~48 GiB (профили и служебные индексы при
100 млн MAU). Ожидаемая нагрузка: ~100 RPS средняя, ~500 RPS пиковая
(read/write). Целевая задержка p95: <30 ms чтение, <50 ms запись.
. Сервис Заказов
.. Тип: PostgreSQL. Назначение: хранение заказов, статусов и агрегатов оплаты.
Необходимый объем: ~8.9 TiB (рост ~1.7 TiB/год, сжатие ×2, горизонт 10 лет);
партиционирование по дате заказа разделяет горячие и холодные данные,
партиции старше 3 месяцев переносятся в tablespaces на диски с ZFS.
Ожидаемая нагрузка: ~35 RPS средняя, ~165 RPS пиковая (read/write).
Целевая задержка p95: <80 ms чтение, <120 ms запись.
. Сервис Товаров (CQRS)
.. Master. Тип: PostgreSQL. Назначение: запись каталога и остатков.
Необходимый объем: ~0.5 GiB. Ожидаемая нагрузка: ~35 RPS средняя, ~165 RPS
пиковая (write). Целевая задержка p95: <120 ms запись.
.. Read-replica. Тип: PostgreSQL. Назначение: чтение карточек и листингов
товаров при read-through кэше Redis. Необходимый объем: ~0.5 GiB на реплику.
Ожидаемая нагрузка на replica: ~500 RPS средняя, ~2500 RPS пиковая (read)
при cache hit-rate порядка 70%. Целевая задержка p95: <60 ms чтение.
.. Управление репликацией. Стратегия: физическая. Failover/консенсус
обеспечивает CNPG.

==== NoSQL и кэши

[arabic]
. Сервис Корзины — Cassandra
.. Тип: Cassandra. Назначение: хранение корзин с TTL для короткоживущих
сессий, consistency level LOCAL_QUORUM. Необходимый объем: ~96 GiB логически и
 ~287 GiB физически при RF=3 (оценка ~1 KiB на корзину, 100 млн MAU).
 Ожидаемая нагрузка:~560 RPS средняя, ~2800 RPS пиковая.
(read/write по ключу корзины). Целевая задержка p95: <20 ms.
. Сервис Товаров (query) — Redis (read-through)
.. Тип: Redis. Назначение: кэш карточек и листингов товаров перед
Read-replica PostgreSQL.
Стратегии: Read-Through и Write-Aside + TTL для инвалидации аномальных записей.
Необходимый объем: ~4 GiB (горячие ключи карточек,
листингов и служебный оверхед). Ожидаемая нагрузка: ~1900 RPS средняя,
~9500 RPS пиковая (cache get/set). Целевая задержка p95: <10 ms hit,
<30 ms miss.
. Сервис Проверки Адресов — Redis
.. Тип: Redis. Назначение: кэш результатов валидации/нормализации адресов.
Необходимый объем: ~2-4 GiB (ключи, значения, TTL, служебный оверхед).
Ожидаемая нагрузка: ~180 RPS средняя, ~900 RPS пиковая (cache get/set).
Целевая задержка p95: <10 ms hit, <30 ms miss.

==== Поисковый индекс

[arabic]
. Сервис Товаров — Elasticsearch
.. Тип: Elasticsearch. Назначение: хранение поискового индекса карточек
товаров для полнотекстового поиска и сортировок. Необходимый объем:
~1-2 GiB (индекс и служебные структуры). Ожидаемая нагрузка: ~1040 RPS средняя,
~5200 RPS пиковая (search). Целевая задержка p95: <150 ms.

==== Медиа-контент

Тип: CDN + origin object storage. Назначение: хранение и доставка медиа-файлов
(картинки и миниатюры карточек). Необходимый объем: ~41 GiB/год, ~410 GiB за
10 лет; горячий набор ~147 GiB. Ожидаемая нагрузка: CDN ~6.4 GiB/s средняя,
~31 GiB/s пиковая; origin при hit-rate CDN 0.99 ~76 MiB/s средняя,
~370 MiB/s пиковая. Целевая задержка p95: <100 ms через CDN, <250 ms через
origin.

==== Требования к защите персональных данных

Все хранилища, содержащие ПДн граждан РФ, физически располагаются
на серверах в Российской Федерации.

По запросу пользователя-субъекта ПДн производится каскадное удаление во
всех хранилищах, включая реплики и резервные копии.
Данные заказов проходят псевдонимизацию.

Подтверждение удаления фиксируется в аудит-логе.
